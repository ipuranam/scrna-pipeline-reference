scRNA-seq Pipeline: End-to-End Reference

From sample prep to sequencing to analysis — a complete guide to the decisions, failure modes, and QC checkpoints in multiplexed single-cell RNA-seq.

Why Single-Cell

Bulk RNA-seq gives you the average. Single-cell gives you the distribution.

Bulk RNA-seq measures gene expression averaged across millions of cells — meaning rare cell types, transient states, and cell-to-cell heterogeneity are invisible. Single-cell RNA sequencing (scRNA-seq) captures the transcriptome of individual cells, revealing the composition and state of every cell type in a sample. This is what makes it powerful for drug discovery, immunology, oncology, and developmental biology — but it also introduces a set of technical challenges that don't exist in bulk workflows.

How It Works

Our scRNA-seq workflow — Illumina Single Cell 3′ RNA Prep, T100 (14-split)

We use Illumina's Single Cell 3′ RNA Prep platform with T100 kits split into 14 PIP reactions, capturing ~500 cells per sample across 20-sample multiplexed pools using Particle-templated Instant Partitions (PIPs). Unlike microfluidics-based systems, PIPs are created by vortex mixing — cells are combined with barcoded particles to form templated emulsions on the benchtop. Inside each partition, the cell lyses, its mRNA binds to the barcoded bead, and reverse transcription creates barcoded cDNA. After breaking the emulsion, cDNA is pooled, amplified, and sequenced. HTO libraries are prepared separately using the SNT (Synthetic Nucleotide Tag) Enrichment kit.

Cell Hashing

What are HTOs and why do we use them?

We use BioLegend TotalSeq-A hashtag antibodies to label cells from different samples before pooling. TotalSeq-A antibodies are conjugated to unique DNA barcodes (Hashtag Oligonucleotides, or HTOs) that target ubiquitous surface proteins. Each of our 20 samples per pool gets a distinct HTO barcode. After sequencing, the HTO signal determines which sample each cell came from. Because TotalSeq-A uses a poly-A tail design, HTO-derived cDNA is captured alongside mRNA during reverse transcription — the SNT Enrichment kit then selectively enriches and amplifies these HTO sequences into a separate library for sequencing.

The Central Problem: Doublet vs. HTO Cross-Contamination

True Doublet

Two cells, one droplet

  • HIGH signal for both HTO-1 and HTO-2
  • Hybrid / chimeric transcriptome
  • Elevated UMI and gene count
  • Falls between clusters on UMAP
✓ Correctly called as doublet — must be removed
HTO Cross-Contamination

One cell + ambient HTO noise

  • High HTO-1, low background HTO-2
  • Clean single-cell transcriptome
  • Normal UMI and gene count
  • Clusters normally on UMAP
✗ Falsely called as doublet — good data lost!
The Core Problem

Both scenarios produce multi-HTO signal — demux tools can't always tell them apart

When HTO labeling quality is high, the distribution of counts for each HTO is cleanly bimodal: cells from a given sample have very high counts ("positive"), and all other cells have near-zero counts ("negative"). Demux tools draw a threshold between these two peaks. But when contamination pushes the negative peak upward, or makes it broader, the positive and negative distributions start to overlap. Now the algorithm can't distinguish a true doublet (two high signals) from a contaminated singlet (one high signal + elevated noise). The result is that perfectly good singlets get thrown out as doublets, and your effective cell recovery rate tanks.

Explore the tabs above to understand every upstream factor that controls contamination severity, and the downstream tools and diagnostics for dealing with it.

Upstream Impact

Sample prep determines the quality ceiling of everything downstream

Fixation method, dissociation protocol, cell viability, and timing decisions all create constraints that no computational method can fully overcome. Poor prep → more ambient RNA/HTO → more false doublet calls → more data loss. This is the highest-leverage point in your pipeline.

Prep Artifacts

Other sample prep factors that propagate downstream

⚠

Dissociation-induced gene expression. Enzymatic dissociation (collagenase, trypsin, TrypLE) triggers immediate-early stress genes: FOS, JUN, HSP90, DUSP1, EGR1. These can dominate variable gene selection and create artificial clusters or cell states. Cold-active protease protocols or computational removal (e.g., filtering known dissociation genes before HVG selection) can mitigate this. Severity is tissue-dependent.

⏱

Time from dissociation to fixation/loading. Cell viability drops with time. More dead cells = more lysis = more ambient RNA and HTO in solution. With DSP-MeOH fixation, aim to fix promptly after dissociation — once fixed, cells are stable at −20°C for up to 7 days. For fresh samples going directly to PIPs, aim for <90 minutes total handling time. Track viability with trypan blue or automated counter and record it — this metadata explains downstream QC variation.

🧫

Cell straining and filtering. Passing cells through a 40µm strainer removes clumps and debris that would become doublets, but can also selectively remove large cell types (e.g., some neurons, adipocytes, megakaryocytes). This biases your cell-type proportions and is a "hidden" data loss — the cells you never see in your data.

🧪

Dead cell removal. MACS dead cell removal kits or DAPI/PI FACS sorting can dramatically improve viability (and therefore reduce ambient RNA/HTO contamination), but they add time, cell loss, and potential bias. The tradeoff: spending 30 minutes on dead cell removal improves data quality but extends handling time and loses 20-40% of total cells.

🧊

Temperature during prep. Keeping cells on ice slows transcription and degradation but also slows enzymatic dissociation. Some protocols use cold-active proteases (psychrophilic enzymes) at 4-6°C to dissociate without activating stress responses. Room temperature processing is faster but induces more artifacts.

Connection

How sample prep choices cascade into your pipeline

Every downstream problem traces back to prep

→

Fixation → HTO signal quality. Methanol denatures surface proteins → HTO antibodies bind poorly → lower positive signal, higher background noise → demux tools misclassify more singlets as doublets. If you're seeing high "doublet" rates on methanol-fixed samples, fix the fixation before tuning the algorithm.

→

Viability → ambient contamination. Low viability = more lysed cells = more ambient RNA soup + more free-floating HTO = the exact contamination problem we covered. A sample with 60% viability has dramatically more ambient contamination than one with 90% viability.

→

Clumping → real doublets. DSP-MeOH fixed cells and poorly dissociated tissues can form aggregates that enter PIPs as pre-formed doublets. These are real multi-cell events, not random co-encapsulation — and they happen at rates independent of your loading density calculation. Passing through a cell strainer is recommended but can lose up to 30% of cells.

→

Dissociation artifacts → false biology. If stress-response genes dominate your highly variable gene selection, PCA and clustering are driven by artifact rather than biology. A "cell type" cluster that's really just "cells that got stressed during prep" will survive all your QC and look real on the UMAP.

The meta-lesson: Computational tools can partially correct for ambient contamination (SoupX, CellBender), filter doublets (DoubletFinder, Scrublet), and remove dissociation genes. But none of them fully recover information lost to poor prep. The ROI on wet-lab optimization — better viability, faster handling, appropriate fixation — is almost always higher than the ROI on computational rescue.
Design Choices

Experimental design determines the severity of every downstream problem

Loading density, number of samples per pool, sequencing depth, and controls are all decisions you make before generating a single read — and they set the upper bound on what your analysis can achieve.

The Central Tradeoff

Loading density vs. doublet rate

Doublet rate scales with loading density. More cells per PIP reaction = lower cost per cell, but more multiplets. Our standard loading is 500 cells per sample with 20 hashtags per pool (10,000 total input cells per T100 14-split reaction). At this loading, multiplet rate is low. With 20 HTOs per pool, cross-sample doublets are detectable but within-sample doublets remain invisible.

Multiplexing Strategy

How many samples per pool, and what kind

Genetically Distinct Donors

Pooling samples from different donors gives you an orthogonal demux signal (SNP-based via vireo/demuxlet) alongside HTO. This is the gold standard because genetic demux is independent of HTO quality. You can cross-validate and identify when HTO contamination is causing false calls.

✓ Ground-truth validation available
✗ Requires genetically distinct samples

Same-Donor, Different Conditions

Common in perturbation/treatment experiments. No genetic differences to leverage — you're entirely dependent on HTO signal for demux. This is where HTO contamination hurts most, because there's no independent validation of sample assignments.

✓ Controls for donor variation
✗ No genetic demux fallback — HTO is your only signal

Pool Size (We use 20)

At 20 samples per pool, ~95% of doublets are cross-sample and HTO-detectable. This is near the practical ceiling — beyond 20 HTOs, demux tool accuracy degrades further (GMM-Demux and BFF fail entirely), ambient HTO complexity increases, and per-HTO signal-to-noise drops. Published benchmarks show dramatic performance drops for most tools going from 10 to 30 samples.

✓ 95% of doublets detectable at 20 samples
✗ Most demux tools degrade — only deMULTIplex2/hashedDrops hold up

Sequencing Depth

Our standard is 20,000 total reads per cell (RNA + HTO combined) at 500 cells per sample, yielding 10M reads per sample sequenced on the Element AVITI at ~$1.76/M reads ($17.60/sample). For the SNT-enriched HTO library, ensure adequate read allocation — BioLegend recommends ~100 reads per cell per antibody for TotalSeq-A. With 20 HTOs, that's ~2,000 HTO reads per cell. Under-sequencing the HTO library specifically can degrade demux quality even when the RNA library is well-sequenced.

✓ Deeper sequencing → more genes detected per cell
✗ Diminishing returns past ~70% saturation — allocate budget to HTO library too
Controls

Controls that actually tell you if things went wrong

🐭

Species-mixing (barnyard) control. Pool human + mouse cells. After sequencing, each cell barcode should map to either human or mouse — barcodes mapping to both are doublets. This gives you an empirical doublet rate that's independent of any computational method. Run this when validating new protocols or platforms.

🏷️

HTO-negative control cells. Spike in unstained cells (different species or cell line) as negative controls. These should have zero HTO signal. If they show HTO counts, that's ambient contamination — and the level tells you how noisy your background is. This is exactly how the original Cell Hashing paper validated the method.

📊

Known-composition reference (e.g., PBMCs). Include a well-characterized reference sample in your pool. If your expected cell-type proportions match known values, your pipeline is working. If monocytes are 5% when they should be 15%, you may be losing a population to filtering or doublet removal.

🔬

Viability measurement. Record viability (trypan blue, automated counter) for every sample before and after processing. This single metric predicts ambient contamination severity better than any downstream QC check. Include it in your metadata — you'll need it to explain sample-to-sample variation.

Design principle: Every experiment should include at least one way to independently verify that your doublet rate, HTO demux, and cell-type recovery match expectations. Without controls, you're trusting your pipeline blindly — and as we've covered, the pipeline can hide its own failures from you.
Doublet Classification

Four types of doublets — each with different detectability

The combination of two axes — same vs. different cell type, and same vs. different sample — creates four distinct categories. Your pipeline needs different strategies for each.

Cross-sample Heterotypic

Two different cell types from two different HTO samples. E.g., a T cell (HTO-1) + macrophage (HTO-2).

HTO-detectable Transcriptome-detectable

Easiest to catch — both methods flag it.

Cross-sample Homotypic

Same cell type from two different HTO samples. E.g., two T cells from HTO-1 and HTO-2.

HTO-detectable Transcriptome-invisible

HTO saves you here — RNA-only methods miss it.

Within-sample Heterotypic

Two different cell types from the same HTO sample. E.g., a T cell + B cell both from HTO-1.

HTO-invisible Transcriptome-detectable

Only computational tools (DoubletFinder, Scrublet) can flag these.

Within-sample Homotypic

Same cell type from the same HTO sample. E.g., two T cells both from HTO-1.

HTO-invisible Transcriptome-invisible

Essentially undetectable. Looks like a normal singlet with slightly higher UMI.

Key implication: HTO demultiplexing only catches cross-sample doublets. Even with 10 samples pooled, ~10% of all doublets are within-sample and invisible to hashing. You still need transcriptome-based doublet detection as a second pass.
Wet Lab Sources

Where ambient HTO comes from

  • Free-floating oligos: Unbound HTOs remain in solution after staining and get captured in droplets alongside cells
  • Antibody dissociation: HTO-conjugated antibodies aren't covalently bound — they detach over time, especially with agitation or temperature changes
  • Insufficient washing: The single biggest lever. More stringent washes after staining dramatically reduce ambient HTO
  • Cell death / lysis: Dying cells release bound HTOs into the ambient pool. Low-viability samples are high-contamination samples
  • Non-specific binding: TotalSeq-A hashtag antibodies (anti-CD298/β2M) can bind weakly to cells from other samples, giving them low positive signal for an incorrect HTO. With 20 HTOs per pool, the cumulative non-specific binding from 19 "wrong" hashtags contributes significantly to background noise per cell.
Sequencing Sources

Post-capture artifacts

  • Index hopping: On patterned flow cells (NovaSeq), index sequences can swap between clusters, causing HTO reads to be assigned to wrong cells
  • PCR chimeras: During library amplification, incomplete extension products can re-prime on a different template, creating hybrid molecules
  • Barcode collisions: Rare but possible — two cells assigned the same cell barcode merge into an apparent doublet
Rule of thumb: If your "doublet" rate is 2-3× higher than expected for your loading density, contamination — not real doublets — is almost certainly the driver. With 20 HTOs per pool, expected cross-sample doublet detection should account for ~95% of all doublets — if your tool is calling 20%+ as doublets, the signal is contamination, not real multiplets.
HTO-Specific Mitigations

Reducing HTO contamination at the staining and sequencing level

For broader sample prep considerations (fixation, dissociation, viability), see the Sample Prep tab.

  • Titrate TotalSeq-A hashtag antibodies: BioLegend recommends 500ng total for TotalSeq-A hashtags. Over-saturating creates more free oligo in the ambient pool. With 20 HTOs, the total antibody mass scales — ensure each individual hashtag is at the minimum effective concentration.
  • Extra washes (3× minimum): Each additional wash removes a large fraction of ambient HTO. This is the single highest-impact optimization for HTO signal quality.
  • Minimize time between staining and loading: Antibody dissociation increases with time. Move fast from staining → wash → load.
  • Use unique dual indexes: Mitigates index hopping during sequencing. Critical on NovaSeq / patterned flow cells. See the Sequencing Platforms tab for more.
  • Optimize SNT enrichment: The SNT Enrichment kit amplifies HTO-derived cDNA from the PIPs pellet. Under-amplification leads to low HTO counts and poor signal-to-noise; over-amplification can introduce PCR bias across the 20 HTOs. Follow Illumina's SNT protocol cycling recommendations precisely.

Diagnostic Protocol

How to tell if you have a doublet problem or a contamination problem

1

Plot per-HTO count distributions

For each HTO, plot the log-transformed count density across all droplets. Clean data shows two well-separated peaks: a narrow negative peak near zero and a high positive peak. If the gap between peaks is small, or the negative peak is broad/elevated, contamination is high.

2

tSNE / UMAP of HTO count space

Run dimensionality reduction on the HTO count matrix alone (not RNA). Clean data produces tight, well-separated clusters for each sample. Contaminated data shows diffuse clouds with poor separation. Doublet populations should appear as distinct intermediate clusters — if instead you see a continuous smear, that's contamination.

3

Compare doublet rate to expected rate

At our standard loading (500 cells/sample x 20 samples = 10,000 input cells per T100 14-split reaction), multiplet rates are low. If your demux tool calls 20%+ doublets, contamination is inflating the number, not real multiplets. With 20 HTOs, ~95% of real doublets should be cross-sample and HTO-visible.

4

Check RNA QC of called "doublets"

True doublets should have ~2× the UMI count and gene count of singlets (they contain two cells' worth of RNA). Plot nCount_RNA and nFeature_RNA for your called singlets vs. called doublets. If the distributions overlap heavily, those "doublets" are contamination-driven misclassifications.

5

Cross-validate with genetic demux (if possible)

If samples are from genetically distinct donors, run vireo or demuxlet on the RNA data. Compare assignments: cells that vireo calls singlet-from-donor-A but HTODemux calls doublet are almost certainly good cells with noisy HTO signal. This is the most definitive test.

6

Inspect "doublets" on the RNA UMAP

True heterotypic doublets fall between cell-type clusters or form novel intermediate populations. If your called "doublets" are sitting squarely within existing clusters and co-localizing with singlets, they're misclassified singlets.

The decisive test: If genetic demux and HTO demux disagree → contamination is the problem, not real doublets. If your called "doublets" have normal RNA QC metrics and cluster with singlets → contamination. If the called doublet rate far exceeds the expected rate for your loading density → contamination.
UMI Process

What UMIs do — and what happens when they fail

UMIs (Unique Molecular Identifiers) are short random barcodes (~10-12nt) attached to individual mRNA molecules at reverse transcription. After PCR amplification, all copies sharing the same UMI collapse back to a single count — so your final matrix reflects original molecules captured, not sequencing reads.

If any step in this chain misbehaves, your count matrix is corrupted at the foundation — and everything downstream (normalization, clustering, doublet detection, HTO demux) inherits that error.

Failure Modes

Four ways UMI deduplication can go wrong

UMI Collisions (Undercounting)

UMI space is finite (~1M for 10nt, ~16M for 12nt). Highly expressed genes can exhaust available UMIs, so two different molecules get the same barcode and collapse into one count. Compresses dynamic range at the top end.

↓ Highly expressed genes appear less variable than they are

PCR/Sequencing Errors (Overcounting)

A single base error in a UMI during PCR or sequencing creates what looks like a new molecule. One original mRNA appears as 2+ distinct UMIs. Naive exact-match dedup misses these. Tools like UMI-tools use network/adjacency methods to catch them, but imperfectly.

↑ Inflated counts, especially for abundant transcripts

Chimeric UMIs (Scrambled Mapping)

During PCR, incomplete extension products re-anneal to different templates, creating chimeric molecules: one cell barcode + a UMI from a completely different molecule. Scrambles the UMI-to-molecule mapping in a way that's hard to detect.

↕ Stochastic count corruption across genes

Low UMI Diversity

If bead synthesis has biased UMI composition (not truly random), effective UMI diversity is lower than theoretical maximum. Collisions become much more frequent than expected, especially in high-capture cells.

↓ Systematic undercounting in high-RNA cells
Why This Matters Here

How UMI failures connect to the doublet / HTO problem

Direct impacts on your pipeline decisions

1

Doublet detection relies on UMI counts as a QC signal. True doublets should have ~2× the UMI count of singlets. If PCR errors inflate singlet UMI counts, the gap narrows and this diagnostic loses power. You can't distinguish real doublets from high-RNA singlets.

2

HTO quantification uses the same UMI framework. HTO counts go through the identical capture → RT → PCR → dedup pipeline. If UMI dedup fails for your HTO library, the count matrix feeding your demux algorithm is unreliable at the input level. Inflated background = contamination looks worse than it is.

3

Saturation effects distort your denominator. If UMI collisions are significant, sequencing saturation curves plateau early, per-cell complexity looks artificially low, and normalization is off. This cascades into clustering and everything downstream.

4

Cross-contamination signal gets amplified. If ambient HTO molecules get captured and their UMIs aren't properly deduplicated (error-inflated), a small amount of physical contamination gets multiplied into a larger apparent signal — pushing more singlets over the doublet threshold.

Diagnostics

How to tell if your UMI process is working

Sequencing Saturation Curve

Plot unique UMIs vs reads per cell. Healthy: gradual plateau at 70-90% saturation. Red flag: plateau too early (low complexity or collision-dominated) or no plateau at very high depth (error-inflated diversity).

Reads-per-UMI Distribution

Each real UMI should have multiple reads (PCR copies). If a large fraction of UMIs have only 1 read, those are likely sequencing errors creating phantom UMIs. Healthy libraries center well above 1.

UMI-per-Gene Distribution

Most genes in a single cell should have 1-10 UMIs. If you see genes with suspiciously high counts that don't match biology (e.g., a "lowly expressed" gene with 50+ UMIs), error-driven inflation is happening.

UMI Sequence Composition

Check nucleotide frequency at each UMI position. Should be roughly uniform (25% each). Strong biases indicate non-random bead synthesis, which reduces effective diversity and increases collision rate.

Bottom line: If UMI processing is off, it corrupts everything in a way that looks plausible — counts are just slightly wrong systematically. For the doublet vs. contamination question, it means your UMI count-based heuristics become unreliable, and you must lean harder on orthogonal signals: genetic demux, UMAP position, and HTO count distributions.
The Hidden Data Problem

"When you do transformations on data, a lot of the data can hide in the transformation"

Every processing step in your scRNA-seq pipeline is a lossy compression. Each transformation reduces complex, high-dimensional data into a simpler representation — and quality problems that were clearly visible in the raw data can get absorbed, smoothed over, or projected out. If you only do QC on processed data, you're checking the output of a pipeline that may have already hidden the problem from you.

This is not a bug in any particular tool — it's an inherent property of data transformations. The fix is to do QC at every layer, especially the raw layers where artifacts are still visible.

The Fix

QC at every layer — what to check and where

Raw / Pre-transformation QC

Check these BEFORE processing. Problems visible here get hidden downstream.

FASTQ quality: Per-base Q scores, adapter content, GC distribution, read length — catches sequencing-level issues before anything else
Barcode rank plot (knee plot): Where does the cell/empty boundary fall? Is it sharp or ambiguous? A gradual knee = high ambient RNA = contamination source
Raw HTO count distributions: Plot log-counts per HTO before demux. Bimodal separation quality is visible here but invisible after classification into labels
Reads-per-UMI distribution: Fraction of UMIs with only 1 read (likely errors) vs multiple reads (real molecules). Only visible before dedup collapses everything
Mapping rate per sample: Low mapping in one sample suggests degradation or contamination. Invisible after count matrix generation
Sequencing saturation: UMIs vs reads curve — catches UMI collision problems and library complexity issues before they silently distort your counts
Raw nUMI / nGene distributions: Before normalization compresses the range. True doublets at 2× show clearly here, less clearly after log-transform

Processed / Post-transformation QC

Still important, but some problems are no longer visible here.

Cluster marker genes: Do clusters correspond to known biology? A cluster driven by mitochondrial genes = dead cells that should have been filtered
Doublet scores on UMAP: Useful for heterotypic doublets that form novel clusters — but homotypic doublets and contamination-driven false calls won't stand out
Sample mixing on integrated UMAP: Are samples well-mixed per cluster? But over-integration can force mixing even when real biological differences exist
Normalized expression distributions: Look for unexpected bimodality or skew — but subtle ambient contamination may be absorbed into the "normal" distribution
Differential expression between samples: Real biology or residual batch/contamination effect? Hard to tell after integration has smoothed things
Cluster stability: Does clustering change with different parameters? Fragile clusters may be artifacts — but you can't tell if the artifact is biological or technical at this layer
Connecting the Dots

How this ties back to doublets, HTOs, and UMIs

The specific traps in your pipeline

1

HTO demux labels hide classification confidence. After running HTODemux, you get clean labels: singlet, doublet, negative. But you've lost the underlying count distributions that tell you whether calls were confident or borderline. A cell labeled "doublet" with HTO-1=500, HTO-2=12 is very different from one with HTO-1=500, HTO-2=490 — but they get the same label. Always look at the raw HTO counts before trusting the labels.

2

Log-normalization compresses the doublet signal. Your main diagnostic for true doublets (elevated UMI count) is a raw-data feature. After log-transform, a 2× UMI difference looks like an additive offset of ~0.7 in log-space. After scaling, it might disappear entirely. Check nUMI distributions on the raw counts, not after normalization.

3

Integration absorbs sample-correlated contamination. If Sample A has high HTO contamination (poor viability, insufficient washing), cells from that sample will look systematically different in gene expression (ambient RNA contamination in the transcriptome). Integration methods treat this as "batch effect" and remove it — hiding the fact that Sample A's data is fundamentally lower quality.

4

UMI dedup decisions are invisible in the count matrix. By the time you see your count matrix, every UMI collision, every error-inflated UMI, every chimeric molecule has already been silently resolved (or not). The count matrix looks like clean integers, but each number carries uncertainty from the dedup step that you can only assess at the read level.

The principle: Do QC at the layer where the problem is visible. Raw sequencing metrics catch sequencing failures. Raw HTO counts catch contamination. Raw UMI counts catch doublets and dedup errors. By the time you're looking at a UMAP, most of these problems have been absorbed into the transformations and are no longer diagnosable. If it looks clean on the UMAP, that doesn't mean it is clean — it might mean the pipeline hid the problem from you.

Cascade Effects

How base diversity problems propagate through your pipeline

When base calling quality degrades in the barcode/UMI region due to low diversity, the effects cascade through every downstream step:

Low base diversity in cycles 1-28 → Poor cluster registration, degraded base-calling accuracy in barcode/UMI
Barcode errors → Cells misassigned or lost; chimeric cell barcodes create phantom cells
UMI errors → Inflated molecule counts (overcounting) — the UMI failure mode we covered
Inflated UMI counts → Doublet QC heuristics (2× UMI) lose discriminating power
Index hopping (Illumina patterned FC) → HTO reads assigned to wrong cells — mimics cross-contamination signal
PhiX capacity loss (5-20%) → Fewer usable reads per dollar — need to sequence deeper or accept lower coverage
Practical Implications

What this means for your experimental design

If you're on Illumina

We sequence on the Element AVITI (~$1.76/M reads, ~1B reads/day throughput). PhiX spike-in requirements vary by platform (1% on NovaSeq 6000, 2% on NovaSeq X). Use unique dual indexes — the kit provides 8 index combinations (column 3 of the Illumina Single Cell UD Kit). Do not mix index combinations from the kit with the standalone UD kit in the same lane. Check per-cycle base quality in the barcode/UMI region — Q30 drops here directly increase UMI error rates.

Budget for PhiX waste and validate barcode/UMI cycle quality

If you're on AVITI

You get the base diversity advantage (near-zero PhiX waste) and higher per-base accuracy (Q44). But validate your bioinformatic workflow — DRAGEN Single Cell RNA pipeline expects Illumina-format data. The poly-T homopolymer variation in AVITI Read 1 may need workflow adjustments for optimal alignment. Also check that your TotalSeq-A HTO library performs equivalently on AVITI chemistry.

Higher accuracy foundation, but validate your informatics pipeline

Platform-agnostic checks

Regardless of platform: check per-cycle quality plots (SAV on Illumina, equivalent on AVITI). Specifically look at Q scores during cycles 1-28 (barcode + UMI region). If quality dips here, every downstream analysis inherits that error. This is the rawest QC you can do — and it directly connects to the "raw vs processed" principle.

Per-cycle quality in the barcode/UMI region is the ground truth

The cost equation

AVITI's effective throughput advantage isn't just about per-read cost — it's about usable reads. If Illumina needs 15% PhiX for a low-diversity run, AVITI delivers ~15% more usable reads from the same flow cell. For high-multiplexing experiments where you're already cost-constrained, this adds up fast.

Compare usable reads, not total reads
The connection to everything else: Platform chemistry is the bottom of the stack. Base diversity affects barcode accuracy → UMI fidelity → count matrix quality → doublet detection power → HTO demux reliability. Index hopping is a platform-specific contamination source. PhiX waste is a platform-specific cost factor. When evaluating your pipeline, start at the sequencing level — it's the layer where problems originate and the layer where the "raw vs processed" principle matters most.

Pipeline Nodes

What each step does and why it matters

Ordered upstream → downstream. Everything inside the dashed box is what Snakemake would orchestrate.

1. BCL → FASTQ Conversion

bcl-convert / bases2fastq

What it does: The AVITI sequencer produces raw BCL (Base Call) files — binary signal data indexed by tile, cycle, and lane. This step converts those into FASTQ files, which are human-readable text files containing the actual nucleotide sequences and quality scores for each read. It also separates reads by their sample index (i7/i5), so each library (RNA + HTO) gets its own set of FASTQs.

Why it matters: Everything downstream needs FASTQ input. If index demultiplexing goes wrong here, reads from different pools bleed into each other — unrecoverable. For AVITI specifically, Element provides bases2fastq which handles their native output format. Alternatively, Illumina's bcl-convert works if the data is in standard BCL format.

Output: Paired FASTQ files (R1: cell barcode + UMI, R2: cDNA insert) per library per pool. Plus index reads (I1/I2) for sample-level demux.

2. Cell Barcode + UMI Extraction

STARsolo

What it does: Read 1 (45 cycles) contains the cell barcode and UMI packed in a specific layout defined by the PIPseq chemistry. This step parses R1 to extract the cell barcode (which identifies which partition/cell) and the UMI (which identifies the original mRNA molecule), then error-corrects the barcode against a known whitelist of valid barcodes.

Why it matters: This is the hardest step to replicate outside DRAGEN. PIPseq uses a different barcode geometry than 10x Chromium — you need the correct PIPseq barcode whitelist file from Illumina and must configure STARsolo with the exact barcode position layout. If this is wrong, you get zero valid cells. This is the step that needs to be validated against a known DRAGEN run.

Key risk: The PIPseq barcode whitelist and geometry are not publicly documented the way 10x's are. You need to obtain these from Illumina or reverse-engineer them from DRAGEN output. Once you have them, STARsolo handles the rest.

3. Read Alignment

STAR / STARsolo

What it does: Takes the cDNA insert (Read 2) and maps it to a reference genome (e.g., GRCh38 for human, GRCm39 for mouse). Each read gets assigned to a gene (or marked as unmapped/ambiguous). STARsolo does barcode extraction and alignment in a single step, which is why it's the preferred tool — no separate barcode parsing needed.

Why it matters: Alignment quality directly determines the accuracy of gene expression counts. Poor mapping (wrong genome build, missing gene annotations, or low-quality reads) means lost signal. STAR is splice-aware, meaning it correctly handles reads spanning exon-exon junctions — essential for RNA-seq.

Output: BAM file with cell barcode, UMI, and gene assignment tags per read.

4. Cell Calling (Real vs. Empty)

EmptyDrops / STARsolo knee

What it does: Not every barcode represents a real cell — most partitions are empty, containing only ambient RNA. Cell calling algorithms distinguish real cell-containing partitions from empty ones by looking at the distribution of UMI counts per barcode. The classic approach is the "knee plot" — barcodes are ranked by total UMIs, and there's a sharp drop-off between real cells and empty droplets. EmptyDrops uses a statistical test against the ambient profile for borderline cases.

Why it matters: Aggressive cell calling (too many "cells") means you include empty droplets that are pure ambient RNA — these become noise in clustering. Conservative calling (too few) means you lose real but low-quality cells. At 500 cells/sample with 20 hashtags, you expect ~10,000 total input cells per reaction, so you should recover somewhere in that range.

Output: A filtered list of valid cell barcodes — everything past this point only includes "real" cells.

5. UMI Deduplication + Count Matrix

STARsolo / umi_tools

What it does: PCR amplification creates many copies of each original mRNA molecule. UMI deduplication collapses reads that share the same cell barcode + UMI + gene into a single count, so each molecule is counted exactly once. The result is a count matrix: rows = genes (~20,000-30,000), columns = cells, values = unique molecule counts.

Why it matters: Without deduplication, highly-amplified molecules would appear more abundant than they actually are, distorting expression measurements. The count matrix is the fundamental data structure — everything downstream (clustering, differential expression, RL features) starts from this matrix.

Output: Two count matrices — RNA (gene expression, stored as sparse matrix in MEX or H5AD format) and HTO (hashtag antibody counts per cell). Both are needed for the next step.

6. HTO Demultiplexing

deMULTIplex2 + hashedDrops

What it does: Each cell has counts for all 20 hashtag antibodies. HTO demux assigns each cell to its sample-of-origin by determining which HTO has dominant signal. Cells with two strong HTOs are flagged as cross-sample doublets. Cells with no clear HTO signal are marked as "negative" (possibly empty droplets or failed staining). deMULTIplex2 is the primary tool (robust to noise with 20 HTOs); hashedDrops is used as a secondary validator.

Why it matters: This is what makes multiplexing work — without it, you can't tell which of the 20 samples each cell came from. It's also your primary doublet detection mechanism (see the Doublet Taxonomy tab). Quality of this step depends heavily on HTO staining quality, washing stringency, and sequencing depth of the HTO library.

Output: Per-cell sample assignment (1 of 20 samples, doublet, or negative). The RNA count matrix is now split into 20 per-sample matrices.

7. Quality Control + Filtering

Scanpy / Seurat

What it does: Removes low-quality cells (too few genes, too few UMIs, too high mitochondrial %), removes doublets flagged by transcriptome-based tools (DoubletFinder, scDblFinder), and corrects for ambient RNA contamination using SoupX or CellBender. Also filters out genes detected in too few cells.

Why it matters: Garbage in, garbage out. Low-quality cells distort clustering and make downstream differential expression unreliable. Ambient RNA correction is especially important for HTO-multiplexed data where ambient signal from all 20 samples is present in every droplet.

Output: A clean AnnData object (H5AD file) — cells x genes, with metadata (sample assignment, QC metrics, doublet scores).

8. Normalization + Dimensionality Reduction

Scanpy / scVI

What it does: Raw counts are normalized (library-size correction + log transformation, or model-based via scVI), highly variable genes are selected (~2,000-5,000), PCA reduces the ~20,000-gene space to ~30-50 principal components, and a neighbor graph is built in PC space for clustering and UMAP visualization.

Why it matters: Raw counts are noisy and sparse — cells have different total counts (sequencing depth), and most genes are zeros. Normalization makes cells comparable. PCA captures the dominant sources of variation. The neighbor graph defines which cells are "similar" — this is the foundation of both clustering and UMAP.

Output: AnnData with PCA embeddings, UMAP coordinates, and neighbor graph.

9. Clustering + Cell Type Annotation

Leiden / CellTypist / manual markers

What it does: Community detection algorithms (Leiden or Louvain) partition the neighbor graph into clusters of transcriptionally similar cells. Each cluster is then annotated with a cell type identity — either manually via known marker genes, or automatically using reference-based tools like CellTypist or scType.

Why it matters: This is where biology emerges. Clusters correspond to cell types, cell states, or developmental stages. For iOB's biomaterial screening, this is where you see whether a treatment condition changed the cellular composition — e.g., shifted the ratio of progenitor cells to differentiated cells, or activated a stress response program.

Output: Per-cell cluster labels and cell type annotations. Differential expression results between clusters. UMAP visualizations colored by cluster, sample, condition.

10. Differential Expression + Feature Extraction

Scanpy rank_genes_groups / pseudobulk DE

What it does: Compares gene expression between conditions (e.g., treated vs. control, biomaterial A vs. biomaterial B) within each cell type. Can be done at single-cell level (Wilcoxon, t-test) or pseudobulk level (aggregate cells per sample, then use DESeq2/edgeR — more statistically rigorous for replicated experiments). Also extracts quantitative features: cell type proportions, pathway scores, gene program activity.

Why it matters: This is the answer to "what did this treatment do?" For biomaterial screening, you're comparing 20+ conditions simultaneously — this step produces the ranked list of genes and pathways that distinguish each condition. These features are also what feed into the RL model.

Output: DE gene tables per cell type per comparison. Feature matrices (samples x features) ready for modeling.

Downstream: RL Integration

From Count Matrices to Reinforcement Learning

The scRNA-seq pipeline produces structured, quantitative data that can serve as the observation/state space and reward signal for a reinforcement learning system optimizing experimental conditions.

State / Observation Space

What the RL agent "sees" after each experiment round:

Cell type proportions — % of each cell type per condition (e.g., 40% neurons, 30% astrocytes, 30% progenitors)
Gene program scores — pathway-level activity (differentiation score, proliferation score, stress response)
DE signatures — top N differentially expressed genes per condition, magnitude + direction
Pseudobulk profiles — aggregated expression vectors per sample, suitable for embedding
QC metrics — cell recovery, doublet rate, mitochondrial %, ambient contamination level

Action Space + Reward

What the RL agent controls and optimizes for:

Actions (experimental parameters) — biomaterial composition, growth factor concentrations, culture duration, coating substrate, media formulation
Reward signal — defined by your biological objective, e.g., maximize % target cell type, minimize off-target differentiation, maximize expression of functional maturity markers
Exploration strategy — balance between exploiting known-good conditions and testing novel combinations across the 20-sample multiplexed pool
Episode — one experimental round (culture → harvest → scRNA-seq → analyze → design next round). Each episode produces 20 data points (one per sample/condition in the pool)
Data flow: AnnData (H5AD) from step 10 → Python feature extraction script → feature matrix (CSV or Parquet, samples x features) → RL environment state. The 20-sample multiplexing is powerful here: each "episode" produces 20 observations in parallel, dramatically improving sample efficiency for the RL agent compared to running conditions one at a time.
